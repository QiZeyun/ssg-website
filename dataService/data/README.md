# Data 目录使用指南

> 📖 **本指南面向运营人员**：本文档详细说明了如何通过修改 `data` 目录下的文件来更新网站内容和配置，无需编写代码。

## 📁 目录结构

`data` 目录是网站的**数据源目录**，包含了网站的所有可配置内容。你可以通过修改这个目录下的文件来更新网站内容。

```
dataService/data/
├── content/              # 📝 页面内容（Markdown 文件）
│   ├── zh/              # 中文页面内容
│   │   ├── about.md     # 关于页面
│   │   ├── blog/        # 博客目录（可选）
│   │   └── products/    # 产品目录（可选）
│   └── en/              # 英文页面内容
│       ├── about.md
│       └── ...
├── seo-config.json       # 🔍 SEO 搜索引擎优化配置
└── pricing-config.json   # 💰 产品价格配置
```

## 🎯 快速开始

### 创建新页面（最简单的方式）

1. **创建 Markdown 文件**：在 `content/zh/` 或 `content/en/` 目录下创建 `.md` 文件
2. **填写内容**：在文件开头填写页面信息（Frontmatter），然后写正文内容
3. **保存文件**：保存文件后，联系开发人员重新构建网站即可

**示例**：创建 "服务介绍" 页面

```markdown
---
title: "我们的服务"
description: "了解我们提供的专业服务"
date: "2024-01-10"
---

# 我们的服务

我们提供以下专业服务...

## 服务一

详细介绍...
```

文件保存为 `content/zh/services.md`，构建后访问 `/zh/services` 即可看到页面。

## 📝 一、页面内容管理（content 目录）

### 1.1 什么是 content 目录？

`content` 目录存储了所有可以通过 Markdown 文件创建的页面内容。你可以在这里创建、编辑、删除页面，无需编写代码。

### 1.2 目录结构说明

```
content/
├── zh/              # 中文内容目录
│   ├── about.md     # 关于页面
│   ├── services.md  # 服务页面
│   ├── blog/        # 博客文章目录（可选）
│   │   ├── post-1.md
│   │   └── post-2.md
│   └── products/    # 产品页面目录（可选）
│       └── product-1.md
└── en/              # 英文内容目录（结构相同）
    ├── about.md
    └── ...
```

### 1.3 如何创建新页面？

#### 步骤 1：决定页面位置和文件名

- **文件名 = URL 路径**：文件名（不含扩展名）会成为页面的 URL
- 例如：`about.md` → `/zh/about`
- 例如：`blog/post-1.md` → `/zh/blog/post-1`

**命名建议**：
- ✅ 使用英文和连字符：`about-us.md`、`product-guide.md`
- ✅ 使用小写字母
- ❌ 避免使用中文、空格、特殊符号（除了 `-` 和 `_`）

#### 步骤 2：创建 Markdown 文件

在对应的语言目录下创建 `.md` 文件，例如：

```
content/zh/services.md
```

#### 步骤 3：编写 Frontmatter（页面信息）

在文件开头，用 `---` 包裹的 YAML 格式填写页面信息：

```markdown
---
title: "页面标题（必需）"
description: "页面描述，用于搜索引擎和社交媒体分享"
date: "2024-01-10"
---
```

**必需字段**：
- `title`：页面标题（必需）

**可选字段**（可以省略）：
- `description`：页面描述，用于 SEO 和社交分享
- `date`：发布日期，格式：`"2024-01-10"` 或 `"2024-01-10T12:00:00"`
- `tags`：标签列表，如 `["标签1", "标签2"]`
- `author`：作者名称
- `image`：SEO 图片路径，如 `"/og-image.jpg"`
- `slug`：自定义 URL（如果想使用不同于文件名的 URL）

#### 步骤 4：编写正文内容

使用 Markdown 语法编写正文内容：

```markdown
---
title: "我们的服务"
description: "了解我们提供的专业服务"
date: "2024-01-10"
---

# 我们的服务

我们提供以下专业服务，帮助您实现业务目标。

## 服务一：网站开发

我们的网站开发服务包括...

### 特点

- 响应式设计
- SEO 优化
- 快速加载

## 服务二：内容管理

...

## 联系我们

如果您有任何问题，欢迎[联系我们](/zh/contact)。
```

#### 步骤 5：保存并通知开发人员

保存文件后，联系开发人员重新构建网站。构建完成后，新页面就可以访问了。

### 1.4 字段详细说明

> 💡 **提示**：以下说明适用于所有 Markdown 文件的 Frontmatter（页面信息区域）。每个字段都有其特定用途和使用场景。

---

#### 1. `title`（必需）⭐

**字段类型**：`string`（字符串，必需）

**作用**：页面标题，是页面的核心标识

**使用场景**：

1. **页面显示**：作为页面主标题（`<h1>`），显示在页面顶部
2. **浏览器标签页**：显示在浏览器标签页上（用户看到的第一眼）
3. **SEO 搜索引擎**：用于搜索引擎排名，影响搜索结果的点击率
4. **社交媒体分享**：当用户在微信、微博等平台分享页面时，显示为分享卡片标题
5. **浏览器历史记录**：显示在浏览器历史记录和书签中

**实际显示位置**：
- ✅ 页面顶部（大标题）
- ✅ 浏览器标签页：`关于我们 | Your Company Name`
- ✅ 搜索引擎结果：`关于我们 - Your Company Name`
- ✅ 微信分享卡片：标题区域
- ✅ 浏览器历史记录和书签

**示例**：
```yaml
title: "关于我们"
title: "产品 2024 版 - 全新升级"
```

**最佳实践**：
- ✅ 简洁明了，概括页面核心内容
- ✅ 包含关键词（有利于 SEO）
- ✅ 长度建议：30-60 个字符
- ❌ 避免过长（会被截断）
- ❌ 避免重复（每个页面应该有独特的标题）

**什么时候需要修改**：
- 每次创建新页面时都必须填写
- 页面内容更新时，如果主题变化，也需要更新标题

---

#### 2. `description`

**字段类型**：`string`（字符串，可选）

**作用**：页面描述，用于搜索引擎和社交媒体展示

**使用场景**：

1. **搜索引擎结果**：显示在搜索结果中标题下方（摘要），直接影响用户是否点击
2. **社交媒体分享**：微信、微博、Facebook 等平台分享时显示的描述文字
3. **页面展示**：可以显示在页面标题下方（如果页面需要）
4. **SEO 优化**：搜索引擎用此描述来理解页面内容，影响排名

**实际显示位置**：
- ✅ 百度搜索结果：标题下方的灰色小字
- ✅ 微信分享卡片：标题下方的描述文字
- ✅ 微博分享：卡片中的描述
- ✅ 页面元数据（可选择显示在页面标题下方）

**示例**：
```yaml
description: "了解我们的公司、使命和价值观。我们致力于为客户提供优质产品和服务。"
description: "学习如何优化网站的 SEO，提高搜索引擎排名。本文提供完整的 SEO 优化指南。"
```

**最佳实践**：
- ✅ 长度：**120-160 个字符**（太长会被截断，太短信息不够）
- ✅ 内容：概括页面核心内容，**吸引用户点击**
- ✅ 包含关键词：自然地融入相关关键词
- ✅ 有行动号召：鼓励用户查看详情
- ❌ 避免与标题重复
- ❌ 避免使用"欢迎访问"等无意义的词

**什么时候需要填写**：
- ⭐ **强烈建议每个页面都填写**，对 SEO 和社交分享非常重要
- 特别是重要的业务页面、产品页面、博客文章

**不填会怎样**：
- 搜索引擎会使用默认描述或自动截取页面内容（效果不好）
- 社交分享时没有吸引人的描述，分享效果差

#### 3. `date`（发布日期）

**字段类型**：`string`（字符串，可选，ISO 8601 格式）

**作用**：记录页面/文章的发布日期

**使用场景**：

1. **页面展示**：显示在页面上，告诉用户内容发布时间（特别是博客文章、新闻）
2. **内容排序**：按日期排序内容列表（最新优先或最旧优先）
3. **时间线展示**：如果有时间线功能，用于排序
4. **Sitemap 生成**：搜索引擎通过日期判断内容的新鲜度
5. **用户感知**：用户看到发布日期，判断内容是否过时

**实际显示位置**：
- ✅ 页面元数据区域（标题下方）
- ✅ 博客文章列表（如果有）
- ✅ Sitemap.xml（告诉搜索引擎）

**格式**：
```yaml
date: "2024-01-10"              # 简单日期（推荐）
date: "2024-01-10T12:00:00"     # 带时间的日期
date: "2024-01-10T12:00:00+08:00"  # 带时区的日期
```

**显示效果**（自动根据语言格式化）：
- 中文：`2024年1月10日` 或 `2024年1月10日 12:00`
- 英文：`January 10, 2024` 或 `January 10, 2024 12:00 PM`

**示例**：
```yaml
date: "2024-01-10"
date: "2024-01-15T14:30:00"
```

**什么时候需要填写**：
- ⭐ **博客文章**：必须填写，体现时效性
- ⭐ **新闻/公告**：必须填写，告知发布时间
- ⭐ **产品发布**：填写产品发布时间
- ⚠️ 普通页面（如"关于我们"）：可以不填，但如果页面有版本概念，建议填写

**最佳实践**：
- ✅ 使用标准格式：`YYYY-MM-DD`
- ✅ 及时更新：每次修改重要内容时，也应该更新日期
- ❌ 避免未来日期（除非确实是预约发布的内容）

---

#### 4. `lastModified`（最后修改日期）

**字段类型**：`string`（字符串，可选，ISO 8601 格式）

**作用**：记录页面内容的最后修改时间

**使用场景**：

1. **页面展示**：显示在页面底部，告诉用户内容最近更新时间
2. **Sitemap 更新**：告诉搜索引擎内容已更新（有利于重新抓取）
3. **版本控制**：帮助追踪内容的修改历史
4. **用户信任**：用户看到最近更新，知道内容维护良好，值得信赖

**实际显示位置**：
- ✅ 页面底部（如果设置了且与 `date` 不同）
- ✅ Sitemap.xml（作为 `lastModified` 字段）

**格式**：与 `date` 相同

```yaml
date: "2024-01-10"
lastModified: "2024-01-15"  # 15 天后更新了内容
```

**显示效果**：
页面底部会显示：`最后更新: 2024年1月15日`

**什么时候需要填写**：
- ⭐ **内容更新时**：每次修改页面重要内容，都应该更新 `lastModified`
- ⭐ **定期维护**：即使内容没变，定期更新也可以告诉搜索引擎内容活跃
- ⚠️ 如果 `lastModified` 与 `date` 相同，不会显示（因为无意义）

**最佳实践**：
- ✅ 只在与 `date` 不同时填写（否则不显示）
- ✅ 每次重要更新后都更新此字段
- ❌ 不要设置为未来日期

**注意**：
- 如果只填写了 `date`，系统会自动使用 `date` 作为 `lastModified`
- 只有当 `lastModified` 与 `date` 不同时，才会在页面底部显示

#### 5. `slug`（自定义 URL 路径）

**字段类型**：`string`（字符串，可选）

**作用**：自定义页面的 URL 路径，覆盖默认的文件名路径

**默认行为**：如果不设置 `slug`，则使用文件名（不含扩展名）作为 URL

**使用场景**：

1. **URL 优化**：文件名是 `about-us.md`，但希望 URL 是 `/about`（更简洁）
2. **保持旧 URL**：文件重命名了，但希望 URL 保持不变（避免 404）
3. **SEO 优化**：使用更友好、包含关键词的 URL
4. **多语言 URL 统一**：不同语言的文件名不同，但 URL slug 可以统一

**URL 生成规则**：

| 文件路径 | 未设置 `slug` 时的 URL | 设置 `slug` 后的 URL |
|---------|---------------------|-------------------|
| `zh/about.md` | `/zh/about` | - |
| `zh/about-us.md` | `/zh/about-us` | `slug: "about"` → `/zh/about` |
| `zh/blog/post-1.md` | `/zh/blog/post-1` | `slug: "first-post"` → `/zh/first-post` |
| `zh/products/product-2024.md` | `/zh/products/product-2024` | `slug: "product-2024"` → `/zh/product-2024` |

**示例**：

**示例 1：简化 URL**
```markdown
# 文件：zh/about-company.md
---
title: "关于我们"
slug: "about"  # 使用更简洁的 URL
---
```
访问地址：`/zh/about`（而不是 `/zh/about-company`）

**示例 2：保持旧 URL**
```markdown
# 文件：zh/about-v2.md（文件重命名了）
---
title: "关于我们（新版）"
slug: "about"  # 保持旧的 URL，用户不会遇到 404
---
```
访问地址：仍然是 `/zh/about`（用户无感知）

**示例 3：SEO 友好 URL**
```markdown
# 文件：zh/seo-guide-2024.md
---
title: "SEO 优化指南"
slug: "seo-optimization-guide"  # 包含关键词，更利于 SEO
---
```
访问地址：`/zh/seo-optimization-guide`

**什么时候需要使用**：
- ⭐ **文件名不友好**：文件名包含特殊字符或中文，希望 URL 更简洁
- ⭐ **URL 重定向**：文件重命名后，希望保持旧的 URL（避免 404）
- ⭐ **SEO 优化**：希望在 URL 中包含关键词
- ⚠️ 大部分情况下，文件名已经很友好，不需要设置 `slug`

**最佳实践**：
- ✅ 使用小写字母和连字符：`about-us`、`product-guide`
- ✅ 保持简洁：URL 不要太长
- ✅ 包含关键词：如果有利于 SEO，可以在 URL 中包含关键词
- ✅ 唯一性：确保在同一语言下，`slug` 是唯一的
- ❌ 避免特殊字符：不要使用空格、中文、特殊符号（除了 `-` 和 `_`）
- ❌ 避免太长：URL 太长不友好

**注意**：
- ⚠️ **唯一性要求**：如果两个文件设置了相同的 `slug`，后加载的会覆盖前面的（导致冲突）
- ⚠️ **优先级**：`slug` 设置的优先级高于文件名
- ⚠️ **修改影响**：修改 `slug` 会导致 URL 变化，旧 URL 会 404（除非做重定向）

#### 6. `tags`（标签列表）

**字段类型**：`string[]`（字符串数组，可选）

**作用**：给内容打标签，用于分类、筛选和 SEO

**使用场景**：

1. **内容分类**：用于内容的细粒度分类（一个页面可以有多个标签）
2. **内容筛选**：用户可以通过标签筛选相关内容（例如：筛选所有带有"SEO"标签的文章）
3. **相关推荐**：系统可以根据相同标签推荐相关内容
4. **SEO 关键词**：标签会被用作 SEO keywords，帮助搜索引擎理解内容主题
5. **内容聚合**：可以生成标签页面，展示所有带有特定标签的内容

**格式**：
```yaml
tags: ["标签1", "标签2", "标签3"]
tags: ["SEO", "搜索引擎", "优化"]
tags: ["产品", "2024", "新版本", "旗舰"]
```

**实际效果**：
- ✅ 显示在页面的元数据区域（标签云）
- ✅ 用于 SEO keywords（搜索引擎关键词）
- ✅ 可以点击标签，查看所有相关内容（如果实现了此功能）
- ✅ 用于相关文章推荐（推荐相同标签的文章）

**示例**：

**博客文章标签**：
```yaml
---
title: "SEO 优化完整指南"
tags: ["SEO", "搜索引擎", "优化", "网站", "营销"]
---
```

**产品页面标签**：
```yaml
---
title: "产品 2024 版"
tags: ["产品", "2024", "新版本", "旗舰", "AI"]
---
```

**什么时候需要填写**：
- ⭐ **博客文章**：强烈建议填写，有助于分类和推荐
- ⭐ **产品页面**：建议填写产品特性相关的标签
- ⭐ **重要页面**：想要被搜索引擎更好地理解时
- ⚠️ 普通页面（如"关于我们"）：可以不填

**最佳实践**：
- ✅ **数量控制**：建议 3-8 个标签，不要太多（显得杂乱）
- ✅ **相关性**：标签应该与内容相关，不要堆砌无关标签
- ✅ **一致性**：相同主题的内容使用相同的标签（例如：所有 SEO 相关文章都使用"SEO"标签）
- ✅ **关键词优化**：选择用户可能搜索的关键词作为标签
- ✅ **多语言**：如果有英文版本，英文标签也应该填写
- ❌ 避免过度堆砌：不要为了 SEO 而堆砌大量不相关的标签
- ❌ 避免重复：标签之间不要重复（例如：不要同时使用"SEO"和"seo"）

**标签命名建议**：
- ✅ 使用名词：`"SEO"`、`"产品"`、`"教程"`
- ✅ 使用行业术语：`"人工智能"`、`"云计算"`
- ✅ 使用时间相关：`"2024"`、`"最新"`
- ✅ 使用特性相关：`"免费"`、`"旗舰"`、`"新版本"`

#### 7. `author`（作者名称）

**字段类型**：`string`（字符串，可选）

**作用**：标明内容的作者

**使用场景**：

1. **页面展示**：显示在页面的元数据区域，标明作者
2. **内容归属**：明确内容的版权归属
3. **作者主页**：如果将来有作者主页功能，可以链接到作者页面
4. **内容管理**：帮助运营团队追踪内容来源
5. **信任建立**：显示作者可以增加内容的可信度（特别是专业文章）

**实际显示位置**：
- ✅ 页面元数据区域（标题下方，日期旁边）
- ✅ 显示格式：`作者: 张三` 或 `Author: John Doe`

**示例**：
```yaml
author: "张三"
author: "John Doe"
author: "产品团队"
author: "Marketing Team"
```

**什么时候需要填写**：
- ⭐ **博客文章**：强烈建议填写，特别是个人博客或署名文章
- ⭐ **专业内容**：教程、技术文章等，填写作者增加权威性
- ⭐ **团队内容**：可以填写团队名称（如"产品团队"、"内容团队"）
- ⚠️ 普通公司页面（如"关于我们"）：可以不填，或填写"公司名称"

**最佳实践**：

- ✅ 使用真实姓名或团队名称
- ✅ 如果内容由多人合作，可以填写主要作者或团队名称
- ❌ 避免使用昵称或不规范的名称（除非是个人博客）

#### 8. `image`（SEO 分享图片）

**字段类型**：`string`（字符串，可选）

**作用**：指定页面在社交媒体分享时显示的图片

**使用场景**：

1. **微信分享**：用户将页面分享到微信时，显示的卡片图片
2. **微博分享**：分享到微博时显示的图片
3. **Facebook/LinkedIn 分享**：国外社交平台分享时的图片
4. **搜索引擎展示**：有些搜索引擎在搜索结果中也会显示图片（如 Google）
5. **页面预览**：一些聊天工具（如 Slack、钉钉）分享链接时也会显示图片

**实际显示位置**：
- ✅ 微信分享卡片：大图区域
- ✅ 微博分享卡片：图片区域
- ✅ Facebook/LinkedIn：Open Graph 图片
- ✅ 浏览器预览：部分浏览器会在链接预览中显示

**格式**：
```yaml
image: "/og-image.jpg"                    # 网站根目录下的图片
image: "/images/og-about.jpg"            # 子目录下的图片
image: "https://example.com/image.jpg"   # 外部图片 URL（不推荐）
```

**图片要求**：
- **推荐尺寸**：1200 x 630 像素（2:1 比例）
- **文件格式**：JPG 或 PNG
- **文件大小**：建议小于 1MB（太大加载慢）
- **内容要求**：清晰、有吸引力、与页面内容相关

**示例**：
```yaml
---
title: "关于我们"
image: "/og-about.jpg"  # 关于我们页面的分享图片
---
```

```yaml
---

---
```

**什么时候需要填写**：
- ⭐ **重要页面**：首页、产品页、重要文章，建议都设置图片
- ⭐ **需要分享的页面**：预期会被大量分享的页面（如活动页、产品发布页）
- ⚠️ 普通页面：可以不填，会使用网站默认图片

**最佳实践**：
- ✅ 为每个重要页面创建专属的分享图片
- ✅ 图片中包含页面标题或核心信息
- ✅ 使用品牌色彩和 Logo，保持一致性
- ✅ 图片文字清晰可读（因为图片可能被缩略显示）
- ✅ 文件名要有意义：`og-about.jpg`、`og-product-2024.jpg`
- ❌ 避免使用内容中的截图或随机图片
- ❌ 避免图片太小或模糊

**图片放置位置**：

**不填会怎样**：

### 1.6 文件路径与 URL 映射规则

**基本规则**：文件名（不含扩展名）= URL 路径

| 文件路径 | URL | 说明 |
|---------|-----|------|
| `zh/about.md` | `/zh/about` | 直接映射 |
| `zh/about-us.md` | `/zh/about-us` | 使用文件名 |
| `zh/blog/post-1.md` | `/zh/blog/post-1` | 子目录自动映射 |
| `zh/products/item.md` | `/zh/products/item` | 多级目录支持 |

**自定义 URL**：使用 `slug` 字段

| 文件路径 | `slug` 设置 | 最终 URL |
|---------|------------|---------|
| `zh/about-us.md` | `slug: "about"` | `/zh/about` |
| `zh/products/product-2024.md` | `slug: "product-2024"` | `/zh/product-2024` |

### 1.7 工作原理（简单说明）

**无需理解技术细节，只需要知道：**

1. **构建时处理**：当你修改 Markdown 文件后，开发人员运行 `pnpm build` 命令，系统会：
   - 自动扫描 `content` 目录下的所有 `.md` 文件
   - 解析文件内容（Frontmatter + Markdown 正文）
   - 将 Markdown 转换为 HTML
   - 通过动态路由 `[...slug]` 生成静态页面

2. **自动生成路由**：每个 Markdown 文件会自动生成对应的网页地址
   - 文件：`content/zh/about.md` → 网页：`/zh/about`
   - 文件：`content/zh/blog/post-1.md` → 网页：`/zh/blog/post-1`
   - 所有 Markdown 内容页面都通过 `app/[locale]/[...slug]/page.tsx` 动态路由处理

3. **保留路由**：以下路由是固定路由，不会被 Markdown 文件处理，请不要在 `content` 目录下创建同名的 Markdown 文件：
   - `pricing` - 价格页面（由固定路由 `app/[locale]/pricing/page.tsx` 处理）
   - `contact` - 联系页面（由固定路由 `app/[locale]/contact/page.tsx` 处理）

4. **自动 SEO 处理**：系统会自动从 Frontmatter 中提取 SEO 信息，生成搜索引擎需要的元数据

5. **自动更新 Sitemap**：已发布的页面会自动添加到 `sitemap.xml`，方便搜索引擎收录

## 🔍 二、SEO 配置（seo-config.json）

### 2.1 什么是 SEO 配置？

SEO（Search Engine Optimization）是搜索引擎优化配置。这个文件控制网站如何被搜索引擎（如百度、Google）收录和展示。

### 2.2 文件位置

```
dataService/data/seo-config.json
```

### 2.3 可以修改的内容

⚠️ **注意**：修改 SEO 配置需要一定的技术背景，建议在开发人员指导下进行，或者只修改明显易懂的字段。

#### 2.3.1 全局配置（global）

**基础信息**（可以修改）：

```json
{
  "global": {
    "siteUrl": "https://example.com",           // 网站地址
    "siteName": "Your Company Name",            // 网站名称
    "defaultDescription": "网站描述",           // 默认页面描述
    "defaultKeywords": ["关键词1", "关键词2"],  // 默认关键词
    "defaultOgImage": "/og-image.jpg"           // 默认分享图片
  }
}
```

**修改建议**：
- `siteUrl`：修改为实际网站地址
- `siteName`：修改为公司/网站名称
- `defaultDescription`：修改为网站描述（用于搜索引擎）
- `defaultKeywords`：添加相关关键词
- `defaultOgImage`：修改为默认分享图片路径

#### 2.3.2 页面 SEO 配置（pages）

为特定页面设置 SEO 信息：

```json
{
  "pages": [
    {
      "path": "/about",
      "title": "关于我们",
      "description": "了解我们的公司、使命和价值观",
      "keywords": ["关于", "公司", "使命"],
      "ogImage": "/og-about.jpg"
    }
  ]
}
```

**说明**：
- `path`：页面路径（不需要语言前缀，如 `/about` 而不是 `/zh/about`）
- `title`：页面标题（显示在浏览器标签页）
- `description`：页面描述（显示在搜索结果中）
- `keywords`：关键词列表
- `ogImage`：分享时的图片

### 2.4 工作原理（简单说明）

当搜索引擎（如百度、Google）访问网站时，会读取这个配置文件中的信息，用于：
- 生成搜索结果中的标题和描述
- 生成社交平台（微信、微博）分享时的卡片信息
- 优化搜索引擎排名

### 2.5 常见问题

#### Q1: 修改 SEO 配置后，什么时候生效？

**A**: 修改后需要重新构建网站，并且搜索引擎需要重新抓取（可能需要几天时间）。

#### Q2: 页面已经有 Markdown 文件的 Frontmatter，还需要在这里配置吗？

**A**: 不需要。如果 Markdown 文件中已经设置了 `title` 和 `description`，会优先使用 Markdown 中的配置。这里的配置主要用于没有 Markdown 文件的页面（如首页、定价页、联系页）。

3. 或者修改后等待正式构建发布

### Q3: 修改后多久能生效？

**A**: 
- **测试环境**：联系开发人员构建，通常几分钟内可以完成
- **生产环境**：需要走正式发布流程，可能需要几小时到一天

### Q4: 如何添加新的语言版本？

**A**: 
1. 在 `content/` 目录下创建新的语言目录（如 `ja/` 表示日语）
2. 复制现有语言的文件结构
3. 翻译内容
4. 联系开发人员配置新语言支持

### Q5: Markdown 语法不会用怎么办？

**A**: 可以参考以下资源：
- [Markdown 基础语法](https://www.markdownguide.org/basic-syntax/)
- 或者直接询问开发人员

### Q6: 图片应该放在哪里？

**A**: 图片应该放在 `public/` 目录下（不在 `data` 目录），然后在 Markdown 中使用相对路径引用。具体位置需要询问开发人员。

### Q7: 可以批量导入内容吗？

**A**: 可以。如果有多篇内容需要批量导入，可以：
1. 准备所有 Markdown 文件
2. 批量复制到对应目录
3. 联系开发人员统一构建

### Q8: 如何备份内容？

**A**: 所有内容文件都在 `dataService/data/content/` 目录下，建议定期备份这个目录。可以：
1. 使用 Git 版本控制（如果已配置）
2. 定期复制到其他位置
3. 联系开发人员协助备份
